## [Unreleased]

### Added
- **`spec.storage.existingClaimName`: adopt a pre-existing PVC by name.** When set, the operator mounts the named PVC verbatim and skips PVC creation/reconciliation. `retainPolicy` is ignored (operator never owns a pre-existing PVC). The two storage modes are mutually exclusive; the CRD enforces a one-of CEL rule. Motivated by a production migration where an existing PVC's name didn't match the operator's generated naming convention, causing the operator to provision an empty PVC and requiring manual data recovery.
- **`spec.suspend`: scale an agent to zero without deleting the CR.** When `spec.suspend: true`, the operator sets `replicas: 0` on the Deployment, reports `status.phase: Suspended`, and sets `PodReady=False` with reason `Suspended` (never `Degraded`). The CR, PVC, ServiceAccount, and Service are all retained. Reversible by setting `suspend: false`. Survives GitOps reconciles (the intent lives in the CR). Motivated by a case where the only available mechanism to stop a misbehaving agent was to delete the CR — which also GC'd the PVC. `spec.suspend` provides a clean stop-without-delete knob.
- **Phase 11: webhook removal + cert-manager dependency drop.** The Phase 5 validating admission webhook is deleted and replaced with CRD OpenAPI schema (`MinLength=1`) + `x-kubernetes-validations` CEL expressions on the CRD itself. The operator install path no longer requires cert-manager. Eight prior webhook rules migrated: six are now hard errors via CRDV/schema, the previous `networkPolicy` deny-all warning is promoted to a hard reject (configuration error too risky to allow silently), and the `dashboard.ingress.annotations`-empty advisory moves to docs (cluster-specific; not operator-enforceable). Cleans up: `internal/webhook/v1alpha1/`, `config/webhook/`, `config/certmanager/`, `config/default/{manager_webhook,cert_metrics_manager}_patch.yaml`, the four `replacements:` blocks in `config/default/kustomization.yaml`, the webhook Service + `ValidatingWebhookConfiguration` + Issuer + both Certificate blocks (incl. orphaned `metrics-server-cert`) + `webhook-cert` volume/mount + `--webhook-cert-path` arg in the Helm chart, the cert-manager-prereq comment in `values.yaml`, the `--webhook-cert-*` flags + `ENABLE_WEBHOOKS` check in `cmd/main.go`, and the `webhooks:` block in `PROJECT`. **Upgrade note:** existing installs should run `kubectl -n hermes-operator-system delete secret webhook-server-cert metrics-server-cert --ignore-not-found` to clean up orphaned Secrets left by the deleted cert-manager Certificate resources.

- **Phase 10.6: per-agent self-introspection `Role` + `RoleBinding`.** The operator now creates `hermes-<agent-name>-self` in the agent's namespace, pinned to the agent's own `resourceNames`. Grants exactly two capabilities to the agent's SA: `apps/deployments[<self>]:get,patch` (the latter enables `kubectl rollout restart deployment/<self>` for pod-level replacement, e.g. to remount the PVC after a config.yaml change that needs a fresh boot) and `hermes.k8s.undermountain.cc/hermesagents[<self>]:get` (self-introspection of spec + status). Explicitly NOT granted: `pods:*` (the canonical restart path is the gateway's `/restart` slash command), `list`/`watch` on either resource (can't be resourceNames-scoped, would expose sibling agents), and `delete`/`create`/`update`/`*` (not needed for the workflow). This is a deliberate, narrow bend on the operator's general "RBAC reference-only" invariant (operator creates bindings, never roles): the rules need `resourceNames` pinning to express "this Deployment, not any Deployment," which a cluster-wide Role can't express. ONE Role per agent, hardcoded rules (no user input shapes them) — the existing `spec.rbac.roleBindings[]` flow remains reference-only. Operator's own ClusterRole gains `roles:get,list,watch,create,update,patch,delete` (was previously `bind,escalate` only). Same-namespace Role + RoleBinding carry the agent CR's controller `ownerRef` (legal because same-namespace) so deletion cascades via K8s GC — no finalizer dance needed.
- **Phase 10.6: `terminationGracePeriodSeconds=210` on the agent Pod template.** K8s's default 30s would SIGKILL the pod mid-drain. Upstream's gateway drains active turns up to `agent.restart_drain_timeout` (180s default in `hermes_cli/config.py::DEFAULT_CONFIG`), then closes SessionDB + releases the `gateway.lock` flock + writes the `.clean_shutdown` marker. 210s = 180s drain budget + 30s buffer for K8s + tini (`-g`) + our-entrypoint signal forwarding and the post-drain teardown. Without this, in-flight agent turns get abandoned on every pod replacement and the next boot calls `suspend_recently_active()` on every active session.
- **Phase 10: OpenTelemetry distributed tracing.** Operator emits OTLP gRPC trace spans for every reconcile — root `hermesagent.Reconcile` with child spans per reconcile helper (`Reconcile.PVC`, `Reconcile.ServiceAccount`, `Reconcile.RBAC`, `Reconcile.Deployment`, `Reconcile.Service`, `Reconcile.DashboardService`, `Reconcile.DashboardIngress`, `Reconcile.NetworkPolicy`, `Reconcile.Status`, `Reconcile.DashboardProbe`, `Reconcile.SecretsValidation`, `Reconcile.EnsureFinalizer`, `Reconcile.HandleDeletion`). Every span carries `hermesagent.name` + `hermesagent.namespace` attributes; span Status flips `Ok` / `Error` on the reconcile result. Span events on phase transitions (`Reconcile.PhaseTransition` with `from` + `to`), RBAC-allowlist rejection (`Reconcile.RBACRejected` with `roleName`), and dashboard probe failures (`Reconcile.DashboardProbeFailed` with `error`) — timeline annotations that pinpoint exactly when a transition / failure happened in Jaeger / Tempo / Honeycomb / Grafana Cloud, without needing log correlation. Standard OTel env vars only (`OTEL_EXPORTER_OTLP_ENDPOINT`, `OTEL_SERVICE_NAME`, `OTEL_RESOURCE_ATTRIBUTES`, `OTEL_EXPORTER_OTLP_HEADERS`). Helm value `tracing.otlp.endpoint=""` by default — when unset, `internal/tracing.Init` is a complete no-op (zero overhead, no connection attempt, no startup latency). Malformed endpoint URLs fail fast at operator startup rather than silently dropping spans for the pod's lifetime. 10s dial timeout on connection setup. SIGTERM-bounded shutdown flushes the last batch of spans best-effort within 5s.
- **Phase 9: NetworkPolicy + custom metrics + Grafana dashboards.** Three orthogonal operability features.
  - `spec.networkPolicy.{enabled, ingress, egress, policyTypes}` — operator reconciles a per-agent `networking.k8s.io/v1` NetworkPolicy with direct K8s native-type passthrough (no operator-side abstraction). PodSelector targets the agent's standard labels. Toggling `enabled: false` deletes the resource in-place (ownerRef GC only handles CR-deleted). Webhook warns when `enabled: true` with no ingress AND no egress rules (deny-all is almost always user error). Effective enforcement depends on cluster CNI — Calico/Cilium enforce, kindnet does not.
  - Custom Prometheus metrics on the existing `/metrics` endpoint: `hermes_agent_phase{name, namespace, phase}`, `hermes_agent_pod_ready{name, namespace}`, `hermes_agent_gateway_ready{name, namespace, gateway_type}` (only populated when `spec.dashboard.enabled`), `hermes_agent_dashboard_probe_failures_total{name, namespace}`. Registered with controller-runtime's metrics registry; updated after every status patch; cleaned by the finalizer on CR delete (no ghost gauges). Phase transitions delete the prior phase row so dashboards don't lie ("Ready=1 AND Degraded=1" was the original failure mode).
  - ServiceMonitor opt-in via Helm value `prometheus.serviceMonitor.enabled=true` (requires Prometheus Operator's `monitoring.coreos.com/v1` CRDs). Rendered as a separate Helm template gated on the value — chart installs cleanly on barebones clusters with the default off.
  - Two Grafana dashboards under `dashboards/`: `operator.json` (controller-runtime reconcile/error/queue/latency view) and `agent.json` (per-CR phase, pod ready, gateway state, probe failure rate; `$namespace` + `$name` template variable dropdowns). Import via Grafana UI or wrap in a ConfigMap labeled `grafana_dashboard=1` for kube-prometheus-stack sidecar discovery.
  - E2E coverage: NetworkPolicy resource shape + label-selector verification + toggle-off cleanup; webhook deny-all warning surfaces in `kubectl apply` output; HermesAgent `/metrics` endpoint exposes `hermes_agent_phase{name="..."}` after the operator reconciles a CR. NetworkPolicy enforcement testing is out of scope — kindnet (default kind CNI) does not enforce.
- **Phase 8: E2E test suite + CI integration.** Four new Ginkgo Describe blocks in `test/e2e/e2e_test.go` cover HermesAgent CR lifecycle (reconcile → children → PVC retain on delete), admission webhook (rejection paths for image/llmDefaultProvider/gateways/ingress), pod invariants + exec probe shape, and Phase 7b dashboard sidecar (shareProcessNamespace, /api/status reachable, `gateway_running:true` cross-container, status.gateways[] populated, Ingress annotation passthrough, disable-path cleanup of sidecar+Service+Ingress). CRD install + operator deploy moved from `Describe("Manager").BeforeAll` into suite-level `BeforeSuite`/`AfterSuite` so every Describe runs against the same operator install — previously the Manager block tore down the operator before the new Describes ran. Hermes-agent image is pre-pulled + Kind-loaded in `BeforeSuite` to avoid ~2.5GB cold-pull timeouts during specs. Wired into a new `.github/workflows/test-e2e.yml` GHA workflow running `make test-e2e` on every PR with `actions/cache@v4` for the hermes-agent image tarball (cache key `hermes-agent-image-v2026.4.30`). Local wall time ~7 min on a warm machine, ~12-15 min cold. Replaces the per-phase ad-hoc kind smokes we'd been dispatching as one-off agents.
- **Phase 7b: dashboard sidecar feature.** Opt-in via `spec.dashboard.enabled=true`. When enabled, the operator injects a `dashboard` sidecar container running `hermes dashboard --insecure --host 0.0.0.0 --no-open` against the agent's PVC; creates a dedicated `hermes-<name>-dashboard` Service on port 9119; and optionally provisions an Ingress with full annotation passthrough (`spec.dashboard.ingress.{enabled, host, ingressClassName, tls, annotations}`). Per-gateway runtime state lands in `status.gateways[]` via 30s polling of the dashboard's `/api/status` endpoint — including `state` (`connecting` / `connected` / `retrying` / `fatal` / `disconnected`) and `message` for fatal states. The operator NEVER passes `--tui` (would spawn a second hermes process on shared `$HERMES_HOME`). Top-level `gateway_state == degraded` (forward-compat — v2026.4.30 does not emit it) maps to `Phase=Degraded`; `stopped` / `startup_failed` likewise.
- **`pod.spec.shareProcessNamespace: true` on the agent pod when the dashboard sidecar is enabled.** Upstream `hermes dashboard` uses **PID-based gateway-liveness detection** ([upstream Docker docs](https://hermes-agent.nousresearch.com/docs/user-guide/docker): *"Running it as a separate container is not supported: the dashboard's gateway-liveness detection requires a shared PID namespace with the gateway process."*). K8s containers in the same pod have isolated PID namespaces by default — without this flag the dashboard would never see the gateway PID and would always report `gateway_running: false` even on a healthy agent. `shareProcessNamespace: true` is the canonical K8s fix. The flag is left nil when the dashboard sidecar is disabled (no behavior change for dashboard-off pods).
- **Admission webhook warns on `spec.dashboard.ingress.enabled` without auth annotations.** Without an edge auth layer (nginx `auth-url`, traefik forwardAuth, oauth2-proxy), the dashboard's admin endpoints are reachable to anyone who can solicit the SPA's embedded session token; `/api/status` is unauthenticated unconditionally. The webhook does not block — the warning surfaces in `kubectl apply` output so users opt in deliberately.
- **Pod invariants: `replicas: 1`, `strategy: Recreate`.** Hermes gateway holds `fcntl(LOCK_EX|LOCK_NB)` on `$HERMES_HOME/gateway.lock` — a second hermes process crashloops with `Gateway already running (PID …)`. Replicas is no longer a configurable spec field (it never was, but is now explicitly pinned in the desired Deployment). Strategy=Recreate ensures the old pod releases the flock before the new pod starts. Empirically validated by the co-run spike (2026-05-15).
- **Phase 5: validating admission webhook.** Operator-sdk-scaffolded `ValidatingWebhookConfiguration` enforces cross-field invariants the CRD's OpenAPI schema can't express: `spec.image` required, `spec.llmDefaultProvider` must reference an entry in `spec.llmProviders[].name`, gateway / provider entries can't have empty `type` / `name`. Errors surface as a `field.ErrorList` aggregate so the API server returns each violation by path. Webhook is served via cert-manager-issued TLS — chart now ships an `Issuer` + `Certificate` and the `ValidatingWebhookConfiguration` carries a `cert-manager.io/inject-ca-from` annotation. Webhook envtest spec verifies rejection at admit time, before the reconciler sees the CR. Local `make run` users can set `ENABLE_WEBHOOKS=false` to skip webhook startup.
- **Phase 1 MVP complete.** `HermesAgent` CRD + controller reconciling PVC, ServiceAccount, and Deployment from a single CR. Env-var composition (LLM providers → gateways → user extras → operator-stamped) tested in isolation. Status phase (`Bootstrap` → `Provisioning` → `Ready` / `Degraded`) + `PodReady` condition. Envtest integration test suite (77 % statement coverage). Realistic sample CR and operator install guide (`docs/operator/install.md`). Smoke-tested end-to-end in kind.
- Repository bootstrap with Apache 2.0 license, contributor docs, and OSS infrastructure.
- Container image release workflow: multi-arch builds (amd64, arm64) triggered by semver tags, pushed to ghcr.io with sigstore keyless signing, and kustomize release bundle attached to GitHub Releases.

### Performance
- **Release pipeline: parallel per-arch builds + persistent registry cache (F-1).** Replaced the single QEMU multi-arch build job with two parallel jobs (`build-amd64`, `build-arm64`) joined by a `docker buildx imagetools create` manifest-assembly step. Each job uses `type=registry,mode=max` build cache persisted to ghcr.io.
- **Cross-compile arm64 instead of QEMU-emulating it.** Added `--platform=$BUILDPLATFORM` to the Dockerfile's builder stage so the Go compiler runs natively on the runner's amd64 architecture and cross-compiles for the target arch, instead of running the entire compile inside an emulated arm64 environment. Verified end-to-end on a test release: `build-amd64` 21s, `build-arm64` 3m53s (was 38+ min trending toward 45min timeout), total pipeline ~6m20s including manifest combine + cosign + chart publish. The kubebuilder-scaffolded Dockerfile threads `TARGETARCH` into `go build` correctly but forgets `BUILDPLATFORM` on the FROM, which is a common multi-arch Go build gotcha.

### Fixed
- **Stamp `HERMES_POD_UID` on agent container env (closes session-pod crash-GC).** The Plan A Kubernetes execution backend creates session pods that should be owner-referenced to the agent pod, so K8s garbage-collects them when the agent dies. ownerReferences require BOTH `name` AND `uid`; the operator was already stamping `HERMES_POD_NAME` but not `HERMES_POD_UID`, causing the agent's `DirectProvisioner._owner_reference` to omit the ownerRef entirely (correctly: empty-uid ownerRefs are rejected with HTTP 422). Without GC, crashed agents leak session pods until idle TTL / activeDeadlineSeconds backstops catch them. Adding `HERMES_POD_UID` (downward API `metadata.uid`) closes the chain. (An earlier `spec.env` workaround was rejected by server-side apply as a duplicate key.)
- **Stale `RBACSynced=False` condition after successful reconcile.** `computeStatus` preserved any existing False condition, which meant once an RBAC reconcile failed (e.g., a ClusterRole missing from the allowlist), subsequent successful reconciles couldn't clear it. The CR was stuck reporting failure even after the cluster admin added the role and the operator restarted. Now the success path explicitly drops the False condition before `computeStatus` runs, so RBACSynced=True can be set. Surfaced by Phase 2 kind smoke.
- **Helm chart's `operator.allowedClusterRoles` value now wired into the rendered Deployment.** `hack/template-chart-image.py` extended with a second splice that injects `{{- if .Values.operator.allowedClusterRoles }}\n- --allowed-cluster-roles={{ … }}\n{{- end }}` after the operator-sdk-default `--health-probe-bind-address` arg. `helm install … --set 'operator.allowedClusterRoles=cluster-admin\,admin'` now takes effect on the running operator with no manual patching. Verified end-to-end via kind smoke.
- **Operator's own ServiceAccount can now grant rights it doesn't itself hold.** Added kubebuilder RBAC markers `//+kubebuilder:rbac:groups=rbac.authorization.k8s.io,resources={clusterroles,roles},verbs=bind;escalate` so the generated ClusterRole includes the `bind` + `escalate` verbs. K8s's escalation-prevention check is satisfied by `escalate` (you can grant rights you don't hold), and `bind` is required to reference the role in a new binding. Narrower blast radius than granting the operator cluster-admin outright — the operator can ONLY do RBAC management, not arbitrary cluster operations. The per-agent `--allowed-cluster-roles` install-time allowlist remains the practical guardrail on what bindings agents may request. Verified by Helm-installing the operator clean and reconciling a CR that grants `cluster-admin` — no manual `ClusterRoleBinding` needed.

### Changed
- **Probe model: HTTP → exec.** The operator now generates K8s exec probes (`hermes gateway status | grep -q '✓ Gateway is running'`) on each agent container instead of the HTTP probe targeting `/health/detailed`. Reasons: `hermes gateway run` does not bind an HTTP server, so the prior probe always returned "connection refused" and `status.podReady` never flipped True even on a working agent. The dashboard (separate `hermes dashboard` subcommand) is the only thing that exposes HTTP; Phase 7b ships that as an opt-in sidecar with proper `/api/status` polling. Empirically validated against `nousresearch/hermes-agent:v2026.4.30` in kind. The `hermes gateway status` binary is at `/opt/hermes/.venv/bin/hermes` and always exits 0 — we synthesize the health signal via grep.
- **`status.gateways[]` is now populated only when `spec.dashboard.enabled` (Phase 7b).** Without the dashboard sidecar, the operator has no observable per-gateway runtime data, and the field stays nil. Pod readiness (`status.podReady`) reflects overall health. The 30-second requeue loop the operator used to run for `/health/detailed` polling is removed — pod readiness is now observed via the Deployment controller's watch on the container's exec probe.
- **Chart `webhook.enabled` / `webhook.certManager.enabled` values removed.** They were dead — set unconditionally, never read by any template. Rather than wire them up (which would require splicing 6+ `{{- if }}` fragments into the kustomize-rendered output, including mid-Deployment fragments around the `--webhook-cert-path` arg and the cert volume mount), we made cert-manager an explicit hard prerequisite for chart installs. The webhook is a security control and we don't want a misleading half-toggle. Users on clusters that can't run cert-manager have an escape hatch: install via raw kustomize (`make deploy`) and strip the webhook + cert-manager patches. Documented in `docs/operator/install.md`.

### Known issues / follow-ups
- **Dashboard auth is upstream-side ephemeral; the operator does NOT provision a token.** Verified 2026-05-15 against upstream `nousresearch/hermes-agent:v2026.4.30` (`hermes_cli/web_server.py:74`): the dashboard generates an ephemeral `_SESSION_TOKEN = secrets.token_urlsafe(32)` at process start and injects it into the rendered SPA HTML — only the browser session that loaded the page can talk to protected `/api/` endpoints. Upstream does NOT read any environment variable to override this token, and `/api/status` is unconditionally unauthenticated. The operator therefore does not provision a `Secret` or plumb `HERMES_DASHBOARD_TOKEN` — that would have been dead weight masquerading as a feature. Gating external exposure is the user's responsibility via Ingress annotations (`nginx.ingress.kubernetes.io/auth-url`, oauth2-proxy fronting, traefik forwardAuth, etc.) — see `docs/operator/install.md`.
- **Long-term: migrate the chart from kustomize-render + post-processing to hand-written Helm templates.** Today's `hack/template-chart-image.py` works for point edits (two splices: image fields, `--allowed-cluster-roles` arg). It does not scale to structural YAML transforms — multi-block conditional wrapping, cross-section toggles. When the operator stabilizes (less operator-sdk regeneration churn) and external users push back on the cert-manager hard-requirement, the right answer is hand-written templates with native `{{- if }}` blocks plus a CI drift-check (`helm template ... | diff <(kustomize build ...)`). Tracked here as a placeholder; no work scheduled.
- **Document Hermes memory backends + provide example CRs; do NOT enforce any topology.** Memory backend selection lives in upstream Hermes's `$HERMES_HOME/config.yaml` (`memory.provider`, seeded first-boot) plus per-provider env vars carrying credentials/endpoints — never a CRD field (consistent with the "no Hermes-app fields in CRD" invariant; memory wiring rides `spec.env` bags + first-boot config.yaml seed). Verified against upstream `plugins/memory/*` 2026-05-31. **The eight providers split into three orchestration classes, and they are NOT interchangeable — so the operator must not force any single shape:**
  - **Embedded (zero infra, lives on the agent's own PVC):** `holographic` (a SQLite file at `$HERMES_HOME/memory_store.db`, config.yaml-only, no env), `hindsight` in `HINDSIGHT_MODE=local` (embedded daemon), `byterover` (`$HERMES_HOME/byterover/` context tree — but needs the `brv` CLI baked into the image: a hermes-images base/variant concern, not an operator one). These need NOTHING orchestrated — just the config.yaml seed.
  - **SaaS-only (API key, no infra):** `mem0` (`MEM0_API_KEY`), `retaindb` (`RETAINDB_API_KEY`, + local write-behind SQLite queue on the agent PVC), `hindsight` cloud mode (`HINDSIGHT_API_KEY`). Wire the key via `spec.env` `secretKeyRef`. Done.
  - **Server backends (a deployed workload + its own PVC, OR SaaS):** `openviking` (`OPENVIKING_ENDPOINT` default `http://127.0.0.1:1933` — the localhost default signals it was built to run as a sidecar; tenancy via `OPENVIKING_ACCOUNT`/`OPENVIKING_USER`/`OPENVIKING_AGENT`), `honcho` (`HONCHO_BASE_URL` / `HERMES_HONCHO_HOST`). Each supports three topologies: **(1) sidecar + its own PVC** (second container in the agent pod, separate memory PVC distinct from `hermes-<name>-data`, agent talks over localhost — extends PVC-sovereignty to two volumes per agent; **would need a new optional `spec.memorySidecar` feature, parallel to `spec.dashboard`**); **(2) shared in-cluster Service + one PVC** (one server Deployment+PVC for the namespace, all agents point their endpoint env at the Service, isolated logically by the tenancy triple — **works with the operator today, zero new code**, server deployed as a sibling workload via kluctl/Helm); **(3) external/SaaS** (point at a hosted endpoint, no cluster PVC).
  - **Design principle (settled): the universal mechanism is env-bag + config.yaml seed — it spans all eight providers. A `spec.memorySidecar` helper is purely additive sugar for the openviking/honcho-sidecar crowd; it must never be required or on the critical path, because embedded providers can't be sidecars and SaaS providers can't be self-hosted.** UndermountainCC's own use case is topology (2): one shared multi-tenant OpenViking Service + PVC, agents isolated by the tenancy triple — needs no operator code. Other users may prefer the sidecar; the operator documents both and forces neither.
  - **Work to do:** (a) a how-to per class with env + config.yaml shape + runnable example `HermesAgent` CRs; (b) decide whether `spec.memorySidecar` is worth building or whether topology (2) docs suffice. A prior `memory-backend-crd-proposal.md` draft existed but was lost in branch churn and never committed — supersede it with this scope note.
